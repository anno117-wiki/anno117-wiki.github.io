# ホワイトボード：3セッションでうまく開発できる方法

作成: 2026-06-17 / 家老・侍・忍者の3役合同

---

## 家老

### 1. 役割を明確に分離する
- 家老: 采配・対話・優先順位決定のみ。重い実作業は持たない
- 侍: ビルド・実装・コミット
- 忍者: 調査・検証・データ確認
- 役が空席の場合は「空席の最上位役」を自動で名乗る（人数カウント不要）

### 2. 非同期委譲でボトルネックを作らない
- 家老は委譲したらすぐ次の話題に移る（完了を待たない）
- 完了通知が届いたら要約して殿に報告する
- 侍・忍者は委譲を受けたら即承諾し、完了時に家老へ報告する

### 3. ファイル競合を避ける
- 各セッションは担当ファイルを事前に宣言してから編集
- 同一ファイルへの同時書き込みは禁止（git競合の元）
- whiteboard も役別ファイル（whiteboard_karo / samurai / ninja）で分離

### 4. ピア通信のノイズを減らす
- メッセージは短く・構造化（【誰より誰へ】件名+本文）
- 確認応答は1行で十分（「了解。作業開始」等）
- 長文の調査結果はファイルに書いてパスを共有する

### 5. 殿（人間）の判断を要するものは即座に上申
- セッション間で解決できない競合・仕様の不明点は家老が殿へ確認
- 推測で進まない（CLAUDE.md 確認優先ルールと整合）

### 6. 引き継ぎメモを毎回更新
- セッション終了前に handover-next-session.md を更新
- 「次の一手」を明記して次回の家老が迷わないようにする

### 7. ツール化けへの対処
- PowerShell は単一値を返す形式を使う（多列出力を避ける）
- ファイル実在確認は Glob を使う（PowerShell より信頼性高）
- 値そのものは正しいことが多い。整形崩れに惑わされない

## 侍

### 1. ファイル競合を構造で防ぐ

同一ファイルへの同時書き込みが最大のリスク。役割ごとに担当ファイル範囲を事前に決め、侍は `apps/` と `packages/` の実装域、忍者は `docs-notes/` と `_local/` の調査域、家老は `CLAUDE.md` と `docs-notes/handover-*.md` と分担する。競合が起きる前に構造で防ぐ。

### 2. コミット前にビルド検証を侍が一元担当

`vue-tsc` と `bun run build:site` の通過確認は侍が責任を持つ。家老・忍者がコード変更を委譲してきたら、侍が必ずビルドを通してからコミットする。「目視成功報告」禁止（CLAUDE.md準拠）。

### 3. タスクの粒度は「30分以内に完了できる単位」

大きなタスクを受け取ったら侍が自分でサブタスクに分解し、TaskCreate で追跡する。1タスク1コミットを目標にし、家老への報告粒度も1タスク1報告にする（まとめ報告は混線の元）。

### 4. 非同期で動き、完了通知を即送信

家老から委譲されたタスクは受諾後すぐ着手し、完了したら `send_message` で即報告する。家老を待たせない。自分が詰まったときも早めに「詰まり中」と送信し、家老が次の指示を出せる状態を保つ。

### 5. E2Eは変更後に必ず走らせる

実装完了 = E2E通過。テストを後回しにすると家老・忍者の検証フローが止まる。`/e2e` は侍がローカルで先行実行し、失敗があれば即修正してからコミットする。

### 6. PowerShellの出力は単一値で取る

化け対策として、ファイル確認や行数確認は多列 `Format-Table` を避け `(Get-Item path).Length` 等の単一値で取得する。成否判定を文字列パースに依存しない。

### 7. 役別 docs を起動時に読む

`docs-notes/roles/samurai.md`（整備後）を起動時に必ず読む。家老・忍者の役割範囲を把握した上で動くことで「越境」による競合を防ぐ。共通事項は CLAUDE.md、詳細は役別 docs に分離する設計が整備されたらそちらに従う。

## 忍者

### 1. 作業前に必ず現状確認を行う

実装に入る前に Glob・Grep・Read で対象ファイルの実在・内容・依存関係を確認する。
「あるはず」の思い込みで作業すると、存在しないファイルへの操作や競合が起きる（whiteboard統合の失敗が好例）。

### 2. 調査結果は即座に家老へ伝える

忍者の知見は家老の意思決定に直結する。調査完了後は結論・根拠・推奨案をセットで send_message する。
「調査中」のまま長時間音信不通にならない。

### 3. ファイル競合を避けるため役割別ファイルを使う

同一ファイルへの並列書き込みは競合リスクが高い。
各役が自分専用ファイルに書き（whiteboard_karo / whiteboard-samurai / whiteboard-ninja 等）、統合は1役（忍者）が担う。

### 4. 公式データを最初の根拠とする

建物名・商品名など公式ゲームデータで確認できるものは推測で報告しない。
`_local/anno-official-data/official_master.csv` を一次ソースとし、GUID照合で正確な日本語名を取得してから報告する。

### 5. 「化け」を前提に検証手段を二重化する

家老セッションでのツール出力化けの事例から、ツール返値と実ファイル確認を分離する。
重要な成否は PowerShell の単純出力か殿の目視を最終根拠とし、ツール表示だけを信頼しない。

### 6. 未確定事項は「確認待ち」と明記して止めない

albion_underground_pit の公式名のように確認に時間がかかる項目は「保留・実機確認待ち」と明記して引き継ぎに残し、他の作業を止めない。
不確かな情報をそのまま報告せず、不明は不明と伝える。

### 7. 引き継ぎドキュメントを更新してセッションを閉じる

忍者が担当した調査・検証の結果と残課題を `handover-next-session.md` に追記してからセッションを終える。
次回忍者が同じ調査を繰り返さないよう知見をその場で文書化する。
